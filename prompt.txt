📋 INSTRUCCIONES DE INICIO DE PROYECTO: KidLink (Módulo Web)Contexto del Proyecto:Estoy desarrollando un proyecto de seguridad infantil y asistencia en emergencias llamado KidLink. El objetivo principal es ayudar a reunificar a niños perdidos con sus padres a través de un objeto físico (llavero, pulsera o tarjeta) equipado con un código QR y un chip NFC.El ecosistema completo se compone de dos partes:KidLink-Web (En desarrollo ahora): Una aplicación web pública, ultra ligera y optimizada para dispositivos móviles. Cuando un buen samaritano encuentra a un niño perdido y escanea el QR/NFC, se abre esta web. Su función es doble: mostrar de inmediato la información médica esencial y los botones de llamada de emergencia, y capturar automáticamente la geolocalización (GPS) del dispositivo del usuario para enviarla en tiempo real a los padres.KidLink-App (Desarrollo futuro): Una aplicación móvil nativa exclusiva para los padres (desarrollada en Flutter) donde registrarán a sus hijos, gestionarán la información y escribirán el enlace único en el chip NFC usando el hardware del celular. La app escuchará en tiempo real las alertas de la web.🛠️ Stack Tecnológico del EcosistemaPara evitar cualquier incompatibilidad en el futuro, debes ceñirte estrictamente a estas tecnologías:Base de Datos y Backend: Supabase (PostgreSQL), utilizando autenticación nativa, almacenamiento de archivos (Storage) para las fotos y Supabase Realtime para la comunicación entre la web y la app.Aplicación Web Pública (Este módulo): Next.js 14+ (App Router), React, Tailwind CSS. Desplegado en Vercel.Aplicación Móvil de Padres (Módulo futuro): Flutter con Dart.🗄️ Estructura de la Base de Datos (Supabase)La base de datos ya está inicializada en Supabase con las siguientes tablas, relaciones y políticas de Row Level Security (RLS). Tu código debe consumir y alimentar estas columnas con total precisión:public.perfiles_padres (Vinculada a auth.users mediante trigger)id (UUID, Primary Key, FK a auth.users)nombre (TEXT)telefono_emergencia (TEXT)creado_el (TIMESTAMPTZ)public.ninos_tags (Información del niño vinculada al chip físico)id_tag (UUID, Primary Key, autogenerado) -> Este ID compone la URL: /nino/[id_tag]id_padre (UUID, FK a perfiles_padres)nombre_nino (TEXT)informacion_medica (TEXT) -> Alergias, condiciones, tipo de sangre, etc.contacto_alternativo (TEXT)telefono_contacto (TEXT) -> Teléfono directo guardado en el tag para evitar JOINs lentos en la calleurl_foto (TEXT) -> Enlace a la foto del niño en el Storage de Supabaseactivo (BOOLEAN) -> Si es false, la web no debe mostrar informaciónpublic.alertas_escaneo (Registro automático de ubicaciones)id_alerta (UUID, Primary Key, autogenerado)id_tag (UUID, FK a ninos_tags)latitud (NUMERIC(10,7), puede ser NULL si deniegan GPS)longitud (NUMERIC(10,7), puede ser NULL si deniegan GPS)gps_activo (BOOLEAN) -> TRUE si el usuario dio permiso, FALSE si se denegódispositivo_origen (TEXT) -> User-agent o sistema operativo del celular que escaneófecha_hora (TIMESTAMPTZ, autogenerado)visto (BOOLEAN)🎯 Tarea Actual: Implementación de la Ruta Dinámica de EmergenciaNecesito que generes el código para la página de visualización del niño en la ruta dinámica apps/KidLink-Web/app/nino/[id]/page.js (o .tsx).Esta página es crítica y debe cumplir con los siguientes requerimientos estrictos de UX y lógica:Carga Inmediata y Estado de Carga: Debe ser ultra rápida. Muestra un esqueleto de carga (Skeleton) limpio mientras consulta a Supabase usando el id de la URL.Validación del Tag: Si el id_tag no existe o su columna activo es false, muestra una pantalla de error amigable indicando que el tag no está activo.Flujo Silencioso de Geolocalización (CRÍTICO):Tan pronto como la página comience a cargar, debe solicitar de forma asíncrona el permiso de ubicación del navegador (navigator.geolocation).Caso Exitoso: Si el usuario acepta, captura la latitud, longitud y el dispositivo de origen (iOS/Android/Web) e inserta inmediatamente una nueva fila en alertas_escaneo.Caso Denegado / Error: Si el usuario deniega el permiso o el GPS falla, IGUALMENTE debes insertar la fila en alertas_escaneo con la latitud y longitud en null, gps_activo: false y el dispositivo de origen. Esto es vital para que el padre sepa la hora exacta del escaneo aunque no tenga las coordenadas.Diseño de Interfaz de Emergencia (Tailwind CSS):El diseño debe ser 100% enfocado en móviles, limpio, intuitivo y transmitir urgencia y cooperación.En la parte superior, muestra de forma destacada la foto del niño (url_foto) de forma circular y su nombre.Una sección llamativa con la informacion_medica (Alergias o condiciones críticas).Dos botones gigantes y nativos de acción rápida: "Llamar al Padre" (usando tel:${telefono_contacto}) y "Contacto Alternativo".Si el permiso GPS fue denegado, muestra un banner sutil pero claro en la parte superior pidiéndole al usuario que active el GPS de su celular y recargue la página para ayudar a los padres a localizarlo.Por favor, genera el código estructurado para este componente utilizando las mejores prácticas de Next.js App Router y el cliente oficial de Supabase.
